# Prompt Engineering 笔记

记录提示词工程、角色设定、示例学习、链式思考、提示优化等相关题目和知识点。

## 2026-05-22 Negative prompting 的常见场景

来源：用户提供题目

题目：
> "Negative prompting"（负向提示）常见于哪个场景？

选项：
- A. 文本分类任务
- B. 图像生成任务，用于指定不希望出现的内容
- C. 代码调试任务
- D. 数学推理任务

结论：
正确选项：B

大白话解析：
负向提示就是告诉模型“不要生成什么”。它最常见于图像生成，比如希望生成一张人物照片，但不想出现“模糊、畸形手、多余手指、低清晰度、水印”等内容，就会把这些写进 negative prompt。

易错点：
- 负向提示不是不能用于文本任务，只是最典型、最常见的应用场景是图像生成。
- 文本分类、代码调试、数学推理更常用的是任务说明、约束条件、示例、推理步骤等提示方式。

知识点总结：
- Negative prompting = 明确告诉模型不要生成哪些内容，图像生成里最常见。

## 2026-05-22 Temperature 参数的作用

来源：用户提供题目

题目：
> Prompt 工程中"温度（Temperature）"参数的作用是什么？

选项：
- A. 控制模型的推理深度
- B. 控制输出的随机性，值越高输出越多样，值越低输出越确定
- C. 限制输出的最大长度
- D. 决定模型是否使用工具

结论：
正确选项：B

大白话解析：
Temperature 控制模型生成时的随机程度。温度越低，模型越倾向选择概率最高、最稳妥的词，输出更稳定、更可重复；温度越高，模型越敢选择一些概率没那么高的词，输出更发散、更有创意，但也更容易跑偏。

易错点：
- 推理深度通常和模型能力、推理模式、提示词要求有关，不是 temperature 的主要作用。
- 最大输出长度一般由 max tokens / max output tokens 控制。
- 是否使用工具由工具配置、系统策略和模型决策控制，不由 temperature 决定。

知识点总结：
- Temperature = 控制随机性。低温更稳，高温更多样。

## 2026-05-22 Adversarial prompting 的安全测试目的

来源：用户提供题目

题目：
> "Adversarial prompting"（对抗性提示）在安全测试中的目的是什么？

选项：
- A. 提升模型的推理能力
- B. 主动尝试让模型产生有害输出或绕过安全限制，以发现并修复漏洞
- C. 测试模型的创意写作能力
- D. 提高模型的响应速度

结论：
正确选项：B

大白话解析：
对抗性提示就是用一些刁钻、诱导、绕规则的输入去测试模型，看模型会不会被带偏，产生有害内容，或者绕过原本的安全限制。它在安全测试里的目的不是为了真的作恶，而是为了提前发现系统弱点，然后修复提示词、防护策略、过滤器或模型行为。

易错点：
- 对抗性提示不是提升推理能力的训练方法。
- 它可能涉及创意写法，但目标不是测试创作能力，而是测试安全边界。
- 响应速度属于性能指标，不是 adversarial prompting 的重点。

知识点总结：
- Adversarial prompting = 用恶意或刁钻输入测试模型安全边界，目的是发现并修复漏洞。

## 2026-05-22 区分已知和未知信息的提示技术

来源：用户提供题目

题目：
> 以下哪种技术可以帮助模型在回答时区分"已知"和"未知"信息？

选项：
- A. 提高 Temperature
- B. 在 Prompt 中明确要求模型标注信息来源类型（"基于提供的文档"/"基于我的训练知识"/"不确定"）
- C. 减少 Few-shot 示例
- D. 使用更大的模型

结论：
正确选项：B

大白话解析：
如果希望模型区分哪些内容来自提供的资料、哪些是模型自己已有知识、哪些是不确定的，就要在 Prompt 里明确要求它标注来源类型。这相当于给模型一个回答格式和判断规则，让它不要把所有内容混在一起说。

易错点：
- 提高 Temperature 会增加随机性，不能帮助区分已知和未知，反而可能更不稳定。
- 减少 Few-shot 示例通常会减少引导信息，不一定更可靠。
- 更大的模型可能能力更强，但不能保证它自动标注来源和不确定性；明确的 Prompt 约束更直接。

知识点总结：
- 想让模型区分“文档依据 / 训练知识 / 不确定”，要在 Prompt 中明确要求标注来源类型。

## 2026-05-22 Few-shot prompting 的适用场景

来源：用户提供题目

题目：
> 以下哪种情况最适合使用 Few-shot prompting 而非 Zero-shot？

选项：
- A. 任务非常简单，模型已经充分理解
- B. 任务有特定格式或风格要求，需要通过示例展示
- C. 需要减少 token 消耗
- D. 需要提高推理速度

结论：
正确选项：B

大白话解析：
Few-shot prompting 就是在 Prompt 里放几个示例，让模型照着例子的格式、风格、判断标准来回答。它适合任务有特定格式、特定语气、特殊分类标准、边界不太好用文字讲清楚的时候。

易错点：
- 如果任务很简单，Zero-shot 通常就够了。
- Few-shot 会额外放示例，通常会增加 token 消耗。
- 示例越多，上下文越长，通常不会提高推理速度。

知识点总结：
- Few-shot = 给几个例子让模型模仿，适合有格式、风格、边界要求的任务。

## 2026-05-22 JSON Mode 的主要用途

来源：用户提供题目

题目：
> JSON Mode 在提示词工程中的主要用途是什么？

选项：
- A. 生成更长的文本
- B. 强制模型输出符合 JSON 格式的结构化数据
- C. 提高模型的推理能力
- D. 减少模型的幻觉

结论：
正确选项：B

大白话解析：
JSON Mode 的主要作用是让模型按 JSON 格式输出，方便程序直接解析和使用。例如要求模型返回 `{ "label": "positive", "score": 0.92 }` 这种结构化结果，而不是一大段自然语言。

易错点：
- JSON Mode 管的是输出格式，不是输出长度。
- 它不能直接提高模型的推理能力。
- 结构化输出可能让结果更容易校验，但不等于自动消除幻觉；模型仍可能把错误信息填进合法 JSON 里。

知识点总结：
- JSON Mode = 约束输出为 JSON 结构，方便程序解析。

## 2026-05-22 Instruction Following 能力不足的表现

来源：用户提供题目

题目：
> "Instruction Following"（指令遵循）能力不足的表现通常是什么？

选项：
- A. 模型输出速度变慢
- B. 模型输出忽略了提示中的格式要求、约束条件或特定指令
- C. 模型拒绝回答
- D. 模型输出过长

结论：
正确选项：B

大白话解析：
指令遵循能力就是模型能不能按用户或系统给的要求办事。能力不足时，常见表现是没有按指定格式输出、漏掉约束、忽略“只能回答 A/B/C/D”、没有遵守字数要求、没有执行特定步骤等。

易错点：
- 输出速度慢属于性能问题，不是指令遵循问题。
- 拒绝回答可能和安全策略、权限、内容风险有关，不一定是指令遵循差。
- 输出过长只有在违反了“限制长度”这类明确指令时，才算指令遵循问题；单独看“过长”不是最准确答案。

知识点总结：
- Instruction Following 差 = 模型没有按提示中的格式、约束或具体指令执行。

## 2026-05-22 Prompt Template 的主要优势

来源：用户提供题目

题目：
> 以下哪个是"Prompt 模板"（Prompt Template）的主要优势？

选项：
- A. 提高模型的推理能力
- B. 标准化输入格式，便于批量处理和版本管理，减少手动编写 Prompt 的错误
- C. 降低 API 调用成本
- D. 让模型输出更有创意

结论：
正确选项：B

大白话解析：
Prompt 模板就是把提示词中固定不变的部分写好，把变化的部分做成变量。这样每次只需要填入不同数据，就能批量生成结构一致的 Prompt，减少手写错误，也方便比较不同版本的效果。

易错点：
- Prompt 模板主要解决工程化和一致性问题，不会直接提高模型推理能力。
- 模板本身不一定降低 API 成本；如果模板很长，成本还可能增加。
- 创意输出主要和任务要求、模型、temperature 等有关，不是模板的主要优势。

知识点总结：
- Prompt Template = 固定结构 + 变量填充，用于标准化、复用、批量处理和版本管理。

## 2026-05-22 Constitutional AI 中宪法的含义

来源：用户提供题目

题目：
> "Constitutional AI"（宪法 AI）中的"宪法"是指什么？

选项：
- A. 国家法律条文在 AI 中的应用
- B. 一组指导模型行为的原则性规则，模型用这些规则自我批评和修正输出
- C. 模型训练数据的质量标准
- D. AI 系统的硬件架构规范

结论：
正确选项：B

大白话解析：
Constitutional AI 里的“宪法”不是国家法律条文，而是一组原则性规则，用来约束模型应该如何回答。模型可以根据这些原则检查自己的回答是否安全、有帮助、符合价值约束，然后进行修改。

易错点：
- “宪法”是类比说法，不等于把国家法律直接塞进 AI。
- 它关注模型行为约束和自我修正，不是训练数据质量标准。
- 它和硬件架构无关。

知识点总结：
- Constitutional AI 的 constitution = 指导模型行为的一组原则，用于自我批评和修正。

## 2026-05-22 精确计算题的提示技术

来源：用户提供题目

题目：
> 以下哪种提示技术最适合让模型完成需要精确计算的数学题？

选项：
- A. 直接要求模型给出最终数值
- B. 要求模型写出完整计算步骤，并在每步验证中间结果
- C. 使用角色提示（"你是数学天才"）
- D. 提供大量数学公式示例

结论：
正确选项：B

大白话解析：
需要精确计算时，最好让模型把题目拆成步骤，并检查每一步的中间结果。这样可以降低一步算错还看不出来的风险，也方便发现错误发生在哪一步。

易错点：
- 只要最终数值，模型可能直接猜或中间算错但无法检查。
- “你是数学天才”这类角色提示可能改善语气或风格，但不能保证计算准确。
- 大量公式示例可能有帮助，但如果没有让模型逐步计算和校验，仍然容易出错。

知识点总结：
- 精确计算类任务：要求分步计算 + 验证中间结果，比只要最终答案更可靠。

## 2026-05-22 System Prompt 的最佳设计做法

来源：用户提供题目

题目：
> 在设计 System Prompt 时，以下哪种做法最佳？

选项：
- A. System Prompt 尽量简短，不超过一句话
- B. 明确定义模型的角色、能力范围、行为规范和输出格式要求
- C. 将所有指令放在用户消息中，不使用 System Prompt
- D. 每次对话都随机更换 System Prompt

结论：
正确选项：B

大白话解析：
System Prompt 是给模型设定底层行为规则的地方，适合放稳定、长期有效的要求，比如模型扮演什么角色、能做什么不能做什么、遇到风险怎么处理、输出要遵守什么格式。它不一定越短越好，关键是清晰、稳定、可执行。

易错点：
- System Prompt 不是越短越好；太短可能缺少边界和规范。
- 把所有指令都放用户消息里，会让稳定规则不够明确，也更容易被后续输入冲掉。
- 随机更换 System Prompt 会导致行为不稳定，不适合严肃应用。

知识点总结：
- 好的 System Prompt = 明确角色、能力范围、行为规范、输出格式，并保持稳定。

## 2026-05-22 Chain-of-Thought 的主要优势

来源：用户提供题目

题目：
> Chain-of-Thought（CoT）提示的主要优势是什么？

选项：
- A. 减少模型的 token 消耗
- B. 通过引导模型展示中间推理步骤，提升复杂任务的准确性
- C. 加快模型的推理速度
- D. 提高模型的创意写作能力

结论：
正确选项：B

大白话解析：
Chain-of-Thought 的思路是让模型不要直接跳到答案，而是把中间推理过程写出来。对多步数学题、逻辑题、复杂判断题，这种方式能帮助模型一步步处理问题，提高答案准确性，也方便人检查哪里出错。

易错点：
- CoT 通常会增加输出内容，因此不会减少 token 消耗。
- 因为要写更多推理步骤，通常也不会加快速度。
- CoT 主要服务复杂推理，不是专门提高创意写作能力。

知识点总结：
- CoT = 引导模型写出中间推理步骤，常用于提升复杂任务准确性。

## 2026-05-22 模型输出不稳定时的调整方法

来源：用户提供题目

题目：
> 当模型输出不稳定、每次运行结果差异很大时，最合适的调整是什么？

选项：
- A. 增加模型的 Temperature
- B. 降低 Temperature，或使用更明确的约束指令
- C. 减少输入的 token 数量
- D. 切换到更小的模型

结论：
正确选项：B

大白话解析：
输出不稳定通常说明模型生成时随机性太高，或者 Prompt 约束不够明确。降低 Temperature 可以让模型更倾向于稳定、确定的答案；同时补充明确格式、范围、判断标准，也能减少每次输出差异。

易错点：
- 增加 Temperature 会让输出更随机，通常会更不稳定。
- 减少输入 token 不一定能改善稳定性，除非原输入有大量噪声或冲突信息。
- 更小的模型不一定更稳定，反而可能指令遵循和理解能力更弱。

知识点总结：
- 输出不稳定：优先降低 Temperature，并把 Prompt 约束写清楚。

## 2026-05-22 模型输出过于冗长时的 Prompt 调整

来源：用户提供题目

题目：
> 当模型输出过于冗长时，以下哪种 Prompt 调整最有效？

选项：
- A. 增加 Temperature
- B. 明确在 Prompt 中限制输出长度（如"请用不超过3句话回答"）或要求简洁风格
- C. 减少 Few-shot 示例数量
- D. 切换到更小的模型

结论：
正确选项：B

大白话解析：
如果模型回答太长，最直接有效的方法是在 Prompt 里明确限制长度或要求简洁表达，比如“用不超过 3 句话回答”“只给结论和 2 个要点”“不要展开解释”。模型需要明确知道你要短回答。

易错点：
- 增加 Temperature 控制的是随机性，不是长度。
- 减少 Few-shot 示例可能减少输入长度，但不一定让输出变短。
- 更小的模型不等于更简洁；输出风格仍需要明确约束。

知识点总结：
- 输出太长：直接在 Prompt 中限制句数、字数、段落数或要求简洁风格。

## 2026-05-22 要求模型先分析再给结论的提示结构

来源：用户提供题目

题目：
> 以下哪种提示结构最适合要求模型先分析再给结论？

选项：
- A. 直接问问题
- B. "请先分析...，再给出你的结论..."
- C. "请给出你的结论，然后解释原因"
- D. 提供大量背景信息

结论：
正确选项：B

大白话解析：
如果希望模型按照“先分析、后结论”的顺序回答，就要在 Prompt 中明确写出这个顺序。模型会更容易按你规定的流程组织答案。

易错点：
- 直接问问题没有明确输出顺序，模型可能先给结论，也可能直接展开。
- C 的顺序是先结论再解释，和题目要求相反。
- 背景信息多不代表输出结构清楚；结构要求要明确写出来。

知识点总结：
- 想控制回答顺序，就在 Prompt 中明确写出“先做什么，再做什么”。

## 2026-05-22 In-Context Learning 的定义

来源：用户提供题目

题目：
> 以下哪种描述最准确地定义了"上下文学习"（In-Context Learning）？

选项：
- A. 模型通过微调学习新任务
- B. 模型仅凭 Prompt 中的示例和说明，在推理阶段学会完成新任务，无需更新参数
- C. 模型通过强化学习优化输出
- D. 模型记忆了训练数据中的所有示例

结论：
正确选项：B

大白话解析：
上下文学习指的是模型在当前 Prompt 里看到任务说明和示例后，临时学会怎么做这个任务。这个“学会”只发生在当前上下文中，不会修改模型参数，也不是重新训练模型。

易错点：
- 微调会更新模型参数，和 In-Context Learning 不同。
- 强化学习是训练或对齐方法，不是上下文学习。
- In-Context Learning 不是靠死记训练数据中的所有示例，而是根据当前上下文中的模式进行泛化。

知识点总结：
- In-Context Learning = 推理阶段看 Prompt 中的说明和示例来完成任务，不更新模型参数。

## 2026-05-22 高质量代码输出的提示词写法

来源：用户提供题目

题目：
> 以下哪种提示词写法更容易得到高质量的代码输出？

选项：
- A. "写一个排序函数"
- B. "用 Python 编写一个归并排序函数，要求：有类型注释、有边界处理、有注释说明每个步骤"
- C. "告诉我怎么排序"
- D. "排序算法有哪些？"

结论：
正确选项：B

大白话解析：
高质量代码输出需要明确任务细节。B 指定了编程语言、算法类型、代码质量要求、边界处理和注释要求，模型知道要交付什么样的代码。A 太宽泛，C 和 D 更像解释或科普，不是明确的代码生成任务。

易错点：
- 代码生成提示越具体，越容易得到符合要求的结果。
- 只说“写一个函数”容易导致模型自行假设语言、算法、输入输出和质量标准。
- 询问概念类问题不等于要求产出可用代码。

知识点总结：

大白话解析：
Prompt 在实际项目里应该像代码和配置一样管理。每次改 Prompt，都应该能追踪改了什么、为什么改、效果有没有变好。如果新版本出问题，也能回滚到旧版本。

易错点：
- 只保存最新版本会丢失历史记录，难以排查问题和回滚。
- 硬编码在业务逻辑里会增加维护成本，也不方便实验和版本对比。
- 只保存在本地不利于团队协作、审查和复现。

知识点总结：
- Prompt 版本控制 = 存仓库 + 记录变更原因 + 评估效果变化 + 支持回滚。

- B. 用精确、可量化的语言描述任务要求，避免模糊词汇

- 输出详细不等于具体，具体性强调明确标准和可执行约束。

## 2026-05-23 精确率和召回率的含义

来源：用户提供题目

题目：
> 在评估 Prompt 质量时，"精确率"和"召回率"分别衡量什么？

选项：
- A. 精确率衡量速度，召回率衡量准确性
- B. 精确率衡量正确输出中有多少是真正正确的，召回率衡量所有正确案例中有多少被正确识别
- C. 精确率衡量输出长度，召回率衡量输出多样性
- D. 两者都衡量模型的推理深度

结论：
正确选项：B

大白话解析：
精确率看模型“说是对的/判为目标类的”里面，有多少真的对；召回率看所有本来应该被识别出来的正确案例里，模型找出了多少。前者关注少误报，后者关注少漏报。

易错点：
- 精确率和召回率都不是速度、输出长度、输出多样性或推理深度指标。
- 精确率高不代表召回率一定高；模型很保守时可能误报少，但漏掉很多正确案例。
- 召回率高也不代表精确率一定高；模型全都判为正可能漏报少，但误报很多。

知识点总结：
- 精确率 = 判为正的里面有多少真正；召回率 = 真正该找出的里面找出了多少。

- A/B 测试 = 一次只改一个变量，方便准确归因。

- C. 要求模型列出信息来源后验证
- D. 人工审核模型输出

结论：
正确选项：A

大白话解析：
Chain-of-Verification 的核心是让模型不要生成完答案就结束，而是主动检查自己的答案。流程通常是先给出初始答案，再围绕这个答案提出验证问题，然后独立回答这些验证问题，最后根据验证结果修正原答案。

易错点：
- B 更接近 Self-consistency：多次生成推理路径再投票。
- C 只强调来源验证，不是完整的验证链流程。
- D 是人工审核，不是模型自我验证流程。

知识点总结：
- Chain-of-Verification = 先答题，再提验证问题，独立验证，最后修正答案。

- Zero-shot = 不给示例，直接下任务。

## 2026-06-11 Decomposed prompting 与 Prompt chaining 的关系

来源：用户提供题目

题目：
> "Decomposed prompting"（分解式提示）与"Prompt chaining"的关系是什么？

选项：
- A. 两者完全相同
- B. 分解式提示是一种 Prompt chaining 的设计方法，强调通过专用子提示处理不同子任务
- C. 分解式提示不使用多次 API 调用
- D. Prompt chaining 不需要分解任务

结论：
正确选项：B

大白话解析：
分解式提示就是把一个复杂任务拆成多个更小的子任务，每个子任务用专门的 Prompt 处理。Prompt chaining 则是把这些步骤串起来，让前一步的输出成为后一步的输入。因此，分解式提示可以看作 Prompt chaining 的一种设计方式。

易错点：
- 两者关系很近，但不完全等同。分解式提示强调“拆任务”，Prompt chaining 强调“步骤串联”。
- 分解式提示在工程实现中经常会涉及多次调用，也可能在一次 Prompt 内模拟多个步骤，不能说它一定不使用多次 API 调用。
- Prompt chaining 通常就是为了把复杂任务拆成多个步骤处理。

知识点总结：
- Decomposed prompting = 拆子任务；Prompt chaining = 把子任务步骤串起来执行。

- A. 所有 Agent 共用同一个 System Prompt，保持一致性
- B. 每个 Agent 有明确的职责边界，Prompt 定义其专属能力、输入来源和输出格式
- C. Agent 的 Prompt 越长越好，覆盖所有可能情况
- D. Agent 之间不需要约定交互格式

结论：
正确选项：B

大白话解析：
多智能体系统里，每个 Agent 应该像团队里的不同岗位一样，各自负责清楚的任务。Prompt 需要定义它负责什么、不负责什么、能使用哪些输入、产出什么格式，以及如何和其他 Agent 交接结果。这样才能减少职责重叠、重复工作和输出混乱。

知识点总结：
- Multi-Agent Prompt = 明确每个 Agent 的职责边界、专属能力、输入来源、输出格式和协作接口。

- A. 知识截止日期后的信息模型完全无法处理
- B. 对于知识截止日期之后的信息，需要通过 RAG、搜索工具或在 Prompt 中直接提供最新信息来补充
- C. 只需切换到更新的模型版本即可解决
- D. 通过增加 Temperature 可以让模型"猜测"最新信息

结论：
正确选项：B

- Knowledge Cutoff 之后的信息，要靠 RAG、搜索工具或 Prompt 中提供资料来补充；不要让模型猜。

- A. Fine-tuning 总是优于 Prompt 工程
- B. 当任务可以通过清晰的指令和示例完成时先用 Prompt 工程；当需要大量特定风格/知识/格式的适配时再考虑 Fine-tuning
- C. Prompt 工程只适合简单任务
- D. Fine-tuning 不需要数据，成本更低

知识点总结：
- 选择原则：先 Prompt 工程快速验证和迭代；Prompt 难以稳定适配大量特定模式时，再考虑 Fine-tuning。

## 2026-06-11 创意头脑风暴的 Prompt 设计

来源：用户提供题目

题目：
> 以下哪种 Prompt 设计最适合让模型给出有创意的头脑风暴建议？

选项：
- A. "请给出10个关于X的想法"
- B. "请用发散思维列出20个关于X的创意，包括常规想法和非常规想法，不要自我审查，先数量后质量"
- C. "请给出关于X最好的方案"
- D. "请严格按照逻辑给出X的解决方案"

结论：
正确选项：B

大白话解析：
头脑风暴的目标是先打开思路，而不是一开始就筛出唯一最优方案。B 明确要求发散思维、数量更多、包含常规和非常规想法，并且先数量后质量，这些都能鼓励模型输出更多样、更有创意的结果。

易错点：
- A 太普通，只要求数量，没有明确创意方向和发散要求。
- C 会让模型过早收敛到一个“最好方案”，不适合头脑风暴阶段。
- D 强调严格逻辑和解决方案，更适合分析或决策，不适合发散创意。

知识点总结：
- 创意头脑风暴 Prompt = 发散思维 + 多数量 + 常规/非常规都要 + 先数量后质量。

## 2026-06-11 Least-to-most prompting 的思路

来源：用户提供题目

题目：
> "Least-to-most prompting"（由简到繁提示）技术的思路是什么？

选项：
- A. 先给模型最简单的任务，再逐渐增加难度
- B. 将复杂问题分解，先解决最简单的子问题，再将子问题的解用于解决更复杂的子问题
- C. 从最少的 token 开始，逐渐增加 Prompt 长度
- D. 先给出最少的约束，再逐渐收紧

结论：
正确选项：B

大白话解析：
Least-to-most prompting 是把一个复杂问题拆成一串子问题，先解决最基础、最简单的部分，再把前面得到的结果作为后面更复杂步骤的依据。它强调“子问题之间有依赖关系”，不是简单地随机从易到难做题。

易错点：
- A 只说“逐渐增加难度”，不够准确；关键是前面子问题的答案要服务后面子问题。
- C 是 token 长度控制，不是 Least-to-most。
- D 是约束逐渐收紧，不是由简到繁推理。

知识点总结：
- Least-to-most prompting = 复杂问题拆成依赖子问题，先解简单部分，再逐步解决复杂部分。

## 2026-06-11 Token Budget 管理

来源：用户提供题目

题目：
> 以下哪种说法正确描述了"Token 预算"（Token Budget）管理？

选项：
- A. 只需限制输出的最大 token 数
- B. 合理分配输入中各部分的 token 占比（System Prompt、Few-shot 示例、检索内容、用户输入），确保关键内容不被截断
- C. Token 预算只影响成本，不影响质量
- D. 上下文越长模型效果越好，无需管理 token 预算

结论：
正确选项：B

大白话解析：
Token 预算管理就是安排有限的上下文空间该给谁用。System Prompt、Few-shot 示例、RAG 检索内容、用户输入和模型输出都要占 token。如果不管理，关键规则、重要证据或用户问题可能被截断，导致模型答错、格式乱或遗漏信息。

易错点：
- 只限制输出长度不够，输入上下文也需要预算管理。
- Token 预算不仅影响成本，也影响质量、延迟和上下文是否完整。
- 上下文不是越长越好；太长可能引入噪声，也会挤占关键内容空间。

知识点总结：
- Token Budget = 合理分配 System Prompt、示例、检索内容、用户输入和输出空间，避免关键内容被截断。

- D. 两者完全相同，没有区别

## 2026-06-11 减少模型位置偏差的方法

- A. 增加 Few-shot 示例数量
- B. 在多选项评估任务中随机打乱选项顺序并多次运行，取平均结果
- C. 降低 Temperature 参数
- D. 使用更长的 System Prompt

结论：
正确选项：B

大白话解析：
位置偏差指模型在多选项判断时，可能偏向某些固定位置，比如更容易选第一个、最后一个，或者更偏向某个标签位置。为了减少这种偏差，可以把选项顺序随机打乱，多次运行，再把结果汇总平均，观察模型是否真的看内容，而不是受位置影响。

- 增加 Few-shot 示例数量不一定能解决位置偏差，还可能引入新的示例偏差。
- 降低 Temperature 可以让输出更稳定，但不能根本消除对选项位置的偏好。
- System Prompt 更长不代表偏差更小，关键是评估设计要控制位置因素。

知识点总结：
- 减少 Position Bias = 随机打乱选项顺序 + 多次运行 + 聚合平均结果。

- Prompt 最核心的是指令：先说明模型要完成什么任务。

- C. 模型在 Prompt 中间位置的指令遵循率下降
- D. 多轮对话中模型偏离主题

题目：
> 在不修改模型权重且不接入任何外部知识库的约束下，希望现有大模型迅速按照特定邮件风格生成回复（工期为1天）。应优先采用哪种方法？

选项：
- A. 少样本提示设计（在提示中提供若干风格示例以进行上下文学习）
- B. 指令微调（重新训练部分或全部参数）
- C. 检索增强生成（RAG），在线索引并注入外部文档
- D. 模型蒸馏，将大模型知识迁移到小模型再部署

结论：
正确选项：A

大白话解析：
题目限制了“不改模型权重、不接外部知识库、工期 1 天”，最适合用 few-shot prompting：在提示词里放几封目标风格的邮件示例，让模型模仿格式、语气和表达。

易错点：
- B 需要训练权重，不符合约束。
- C 需要外部知识库，不符合约束。
- D 蒸馏成本高，也不是 1 天内最快适配风格的方法。

知识点总结：
- 不改权重、快速适配风格：优先 few-shot prompt。
